# ADR-204003: Enterprise App Identity and Authentication Flow

| Field | Value |
|---|---|
| **ID** | ADR-204003 |
| **Status** | Accepted |
| **Provider** | Microsoft Azure |
| **Discipline** | Security |
| **Replaces** | ADF-016 |
| **Date** | 2026-06-17 |

---

## Context

Enterprise applications require a consistent, auditable identity and authentication pattern that supports both human users (SSO via Azure Entra ID) and non-human workloads (service-to-service using Managed Identity or service principals). Ad-hoc approaches — hardcoded credentials, shared service accounts, or inconsistent OAuth flows — create audit gaps, privilege escalation risks, and operational fragility.

A unified identity architecture must enforce MFA for all human access, eliminate secrets for workload-to-workload authentication, and provide a single control plane for access review and governance.

---

## Decision

We will standardize on **Azure Entra ID** as the identity plane for all enterprise application authentication:

1. **Human users (internal):** OAuth 2.0 Authorization Code Flow with PKCE + MFA enforced via Conditional Access
2. **Human users (external / B2C):** Azure Entra External ID (B2C) with social or local account providers
3. **Workload-to-workload (Azure-hosted):** Managed Identity with Azure RBAC role assignments — no client secrets
4. **Workload-to-workload (cross-cloud / GitHub Actions):** OIDC Workload Identity Federation — no stored secrets
5. **External service principals:** Client credential flow with **certificate-based** client assertion — no client secrets
6. **Legacy ROPC flow:** Prohibited — bypasses MFA and Conditional Access

---

## Drivers

- Eliminate client secrets for machine-to-machine authentication
- Enforce MFA for all human access via Conditional Access policies
- Provide unified audit log in Entra ID sign-in logs for all authentication events
- Support Zero Trust principle: verify explicitly, use least privilege, assume breach

## Alternatives Considered

| Alternative | Pros | Cons | Reason Rejected |
|---|---|---|---|
| Username/password (ROPC) for services | Simple | Bypasses MFA and Conditional Access; credential stored in plaintext | Prohibited |
| Shared service account + classic PAT | Familiar | Single point of failure; no per-service audit trail; rotation blocks all dependents | Rejected — see [[ADR-204001]] |
| Third-party IdP (Okta, Auth0) | Advanced features, vendor flexibility | Adds identity federation complexity; Azure-native tooling sufficient for current requirements | Revisit at scale |

## Authentication Pattern Reference

| Actor | Flow | Credential Type | MFA Required | Secret Stored |
|---|---|---|---|---|
| Internal employee | Auth Code + PKCE | Federated (SSO / Windows Hello) | ✅ Always (CA policy) | No |
| External partner / B2C | Auth Code + PKCE | Local account or social IdP | ✅ B2C policy | No |
| Azure-hosted workload | Managed Identity | None (platform-issued) | N/A | No |
| GitHub Actions pipeline | OIDC federation | Short-lived JWT | N/A | No |
| External service (API integration) | Client credentials | Certificate | N/A | Certificate in Key Vault |
| Legacy app (temporary) | ROPC | Password | ❌ Bypassed | **PROHIBITED** |

## Consequences

### Positive
- Unified identity plane — all authentication events visible in Entra ID sign-in logs
- Managed Identity eliminates credential rotation burden for all Azure-hosted workloads
- Conditional Access enforces MFA, device compliance, and location-based policies automatically
- Certificate-based service principals are significantly harder to exfiltrate than client secrets

### Negative / Trade-offs
- Managed Identity is Azure-native only — hybrid/on-premises workloads still require service principals with certificates
- App Registration proliferation in large organizations — requires naming convention governance and ownership tagging
- Conditional Access requires Azure AD P1 license minimum (P2 for risk-based policies)

### Risks
- Orphaned App Registrations with active client secrets — enforce quarterly access reviews via Entra ID Access Review feature
- Over-permissioned App Roles assigned at service principal level — audit via `Get-AzureADServiceAppRoleAssignment`
- ROPC flow accidentally enabled — block via Conditional Access policy `Block legacy authentication`

---

## Implementation Notes

- All App Registrations must have `owners` set and a `description` tag with team name — enforce via Azure Policy on `microsoft.aad/applications`
- Conditional Access: create policy `Block Legacy Authentication` (targets all users, all apps, grant: Block)
- Certificate rotation: use Key Vault certificate auto-rotation with `azurerm_key_vault_certificate` + `auto_renew_before_expiry_percent`
- Quarterly access review: configure Entra ID Access Reviews on all App Role Assignments
- Related: [[ADR-204001]] (Token management), [[ADR-204002]] (Bootstrap secret flow)
